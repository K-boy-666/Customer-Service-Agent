# HR 面试回答模板:实现一个最小可用的 Agent

> 适用问题:"实现一个最小可用的 agent,prompt 优化、批量生成、自动评测或者竞品监控等方面"
> 方向推荐:主选 **Prompt 优化 Agent**;备选 **自动评测 Agent**
> 使用方式:先讲框架,再填实战背书,口语化表达,控制在 3-5 分钟

---

## 〇、回答策略(面试前必读)

这道题考的不是"做出多复杂的系统",而是三点:
1. **0→1 的 MVP 落地能力**——能否定义清楚"最小可用边界",不贪大;
2. **对 agent 本质的理解**——agent = LLM + 工具 + 循环 + 评测,缺一不可;
3. **工程化思维**——可验证、可迭代、可监控,而不是"调个 API 跑通就完事"。

回答结构建议用 **STAR + MVP 设计**:
- S/T:一句话定位(做什么、为什么选这个方向)
- A:技术方案(架构图 + 核心模块 + 关键设计取舍)
- R:怎么证明有效(评测指标)+ 实战背书(你的项目经验)

四个方向的选择建议:

| 方向 | 是否推荐 | 理由 |
|------|---------|------|
| Prompt 优化 | ⭐主推 | 最贴合"agent"概念,可做成"分析→优化→验证"闭环;你有 6 个 sub-agent 的 prompt 设计经验背书 |
| 自动评测 | ⭐强备选 | 你的测试体系最硬(84 用例/76% 覆盖/6 阶段验证/5x 加速),差异化最强 |
| 批量生成 | 可选 | 偏内容生产,agent 属性弱,容易做成"调 API 批量跑" |
| 竞品监控 | 可选 | 偏数据采集 + 定时任务,agent 属性弱,工程味重于 AI 味 |

---

## 方向 A:Prompt 优化 Agent(主推·完整模板)

### 一、需求理解与方向选择

> 口语化开头:"这道题我选 prompt 优化方向。"

选这个方向有三个理由:
1. **prompt 是 agent 能力的核心杠杆**——同样的模型和工具,prompt 质量直接决定输出质量,优化 ROI 最高;
2. **它本身可以做成一个 agent**(自动分析 bad case → 生成优化候选 → A/B 验证),形成闭环,符合"最小可用 agent"的定位;
3. **有实战背书**——我在客服智能体项目里设计了 6 个子 agent 的 system prompt,沉淀了通信协议、质量检查清单等可复用经验,这个方向我能做出深度。

### 二、MVP 定义(最小可用边界)

**目标**:给定一个业务 prompt + 一批测试 case(含期望输出),agent 能自动:
- 评估当前 prompt 在测试集上的表现(打分 + bad case 归因)
- 生成优化后的 prompt 候选
- 用 LLM-as-judge 做 A/B 对比,输出胜出的 prompt + 改进报告

**明确不做的(MVP 边界,体现取舍意识)**:
- 不做自动部署上线(人工 review 后采纳,保证安全)
- 不做多轮自动迭代(MVP 只做单轮:分析 → 优化 → 验证)
- 不做 prompt 版本管理完整系统(文件目录 + git 即可)

> 面试加分话术:"MVP 的关键是定义清楚不做什么。我把它收敛到单轮闭环,先验证'自动优化比人工优化的胜率'这个核心假设。"

### 三、技术方案

**架构(单 agent + 工具集 + 评测循环)**:

```
输入: prompt_v0 + test_cases(含 expected 输出)
  ↓
[评测工具] run_prompt_on_cases  → 批量执行,收集实际输出
  ↓
[归因工具] analyze_bad_cases    → LLM 分析失败原因,结构化归类
                                   (指令模糊 / 示例缺失 / 格式不符 / 边界遗漏)
  ↓
[优化工具] generate_improved_prompt → 基于归因 + 优化原则生成 prompt_v1 候选
  ↓
[裁判工具] judge_ab             → LLM-as-judge 盲评 (v0 vs v1),输出胜者
  ↓
输出: prompt_v1 + 评测报告(胜率、bad case 对比、归因统计)
```

**核心模块**:
1. **评测器 Evaluator**:批量执行 prompt on test cases,收集实际输出(确定性执行,不引入 LLM 随机性)
2. **归因器 Diagnoser**:LLM 分析 bad case,输出结构化归因 JSON(失败类型 + 证据原文)
3. **优化器 Optimizer**:基于归因 + 优化原则(清晰性、few-shot 示例、格式约束、边界覆盖)生成新 prompt
4. **裁判 Judge**:LLM-as-judge,用固定评分 rubric 对 (v0, v1) 输出做盲评,输出胜者

**关键设计取舍**:
- 评测用确定性工具执行,只在"归因"和"裁判"环节用 LLM——保证评测可比、可复现;
- 评分 rubric 固定(如 正确性 40% + 完整性 30% + 格式 20% + 安全 10%),避免裁判主观漂移;
- bad case 归因输出 JSON,便于聚合统计"哪类问题最多",指导优化优先级。

### 四、实战背书(引用你的项目经验)

> 这段是差异化关键,用项目数据证明"我能做这件事"。

"我在客服智能体项目里做过类似的 prompt 工程闭环,有几个可直接迁移的经验:
- 为 6 个子 agent 设计了 system prompt,包含**权限声明、通信协议**(输入:客户上下文 + 任务;输出:处理结果 + 客户回复 + 内部备注)、**情绪分级处理策略**、**质量检查清单**(回复前自检 8 项);
- 用**确定性运行时**把 prompt 契约转成可测试的 Python 代码,使路由和工具调用无需启动 LLM 即可测试——这个思路直接迁移到 MVP:用确定性 case 集做基线评测,不依赖 LLM 随机性;
- RAG FAQ 检索做了**双后端**(语义 + 词法)和**阈值调优**(语义 0.25 / 词法 0.04),本质就是'评测-调参'闭环的实战版。"

### 五、评测与迭代(怎么证明有效)

> 面试官必问:"你怎么知道你的 agent 有效?" 这段是硬通货。

MVP 自身的有效性验证:
1. **基线**:固定测试集上,人工优化的 prompt 作为 baseline;
2. **对比**:agent 优化的 prompt vs baseline,用 LLM-as-judge 胜率(要求 > 55% 才算有效);
3. **回归**:优化后的 prompt 不能在原 pass 的 case 上退化(回归测试,防止"按下葫芦浮起瓢");
4. **指标**:bad case 数下降率、平均得分提升、归因准确率(抽样人工校验)。

迭代方向(MVP 之后):
- 多轮自动迭代(直到胜率不再提升,设早停)
- prompt 版本管理 + 灰度上线
- 接入线上真实 case 做持续评测

### 六、工程化

- **测试**:确定性 case 集 + 回归测试(借鉴项目 84 用例 / 76% 覆盖的经验)
- **可观测**:记录每次优化的输入输出、归因、胜率,便于复盘(借鉴项目 structlog 结构化日志 + request_id 链路追踪)
- **成本控制**:评测和归因环节用小模型,裁判用强模型;批量调用控制并发(借鉴项目并发安全经验——我修过 3 个并发 bug:LRU 缓存竞态、SQLite WAL、幂等 TOCTOU)

### 七、价值与延展(收尾)

- **价值**:把 prompt 优化从"人工经验"变成"数据驱动的可复现流程",降低 prompt 维护成本;
- **延展**:同一套"评测-归因-优化-验证"闭环可迁移到**批量生成**(优化生成 prompt)和**竞品监控**(用竞品 prompt 跑评测集对比)——四个方向本质是同一套基础设施。

## 方向 B:自动评测 Agent(强备选·要点模板)

> 你的测试体系最硬,选这个方向差异化最强。把"评测系统"包装成一个 agent。

### 一、定位
"我选自动评测方向。agent 的核心难题不是'能跑通',而是'怎么知道跑得好'。我做一个最小可用的评测 agent,能对任意 agent 的输出做自动化打分 + 归因。"

### 二、MVP 定义
**目标**:给定一个被测 agent + 一批标注 case(含 expected + 评分维度),评测 agent 能自动:
- 批量调用被测 agent,收集输出
- 用 LLM-as-judge + 规则校验混合打分(确定性规则查格式/字段,LLM 查语义质量)
- 输出评测报告:总体得分、分维度得分、bad case 归因、回归对比

**不做的**:不做在线评测(离线批处理即可)、不做多模型横评(MVP 只评单个 agent)

### 三、技术方案
```
输入: 被测 agent 入口 + test_cases(expected + rubric)
  ↓
[执行器]  批量调用被测 agent → 收集实际输出(并发控制 + 超时重试)
  ↓
[规则校验] 确定性检查:格式、字段完整性、安全词、JSON schema
  ↓
[LLM 裁判] 按 rubric 盲评打分(正确性/完整性/格式/安全)
  ↓
[归因聚合] bad case 分类统计 + 回归对比(vs 上一版本)
  ↓
输出: 评测报告(总分、分维度、bad case、回归是否通过)
```

**关键设计**:
- **混合打分**:能用规则确定的(格式、字段、安全)绝不用 LLM;只有语义质量才用 LLM-as-judge——降低成本 + 提高可复现性;
- **rubric 固化**:评分维度和权重配置化,保证跨版本可比;
- **回归门禁**:新版本若在原 pass case 上退化,直接判 fail(借鉴 CI 门禁思路)。

### 四、实战背书(你的最强弹药)
"这块我有完整的实战经验。我的客服智能体项目有一套成熟的评测体系:
- **84 个测试用例,76% 分支覆盖率**,包含端到端、安全控制、RAG 检索、并发(22 用例)、负载(P95 < 2s)多层测试;
- **6 阶段 init 验证**(Python/依赖/DB/REST/MCP/Tests),每次启动自动跑,保证环境一致性;
- **pytest-xdist 并行加速**:81 测试从 63s 降到 13s(5x),用 `--dist=loadscope` 按类分组;
- **harness 结构化验证**:加权评分 100/100;
- **确定性运行时**:把 prompt 契约转成可测试代码,核心路由无需启动 LLM 即可测——这个思路直接迁移到评测 agent:用确定性 case 做基线,LLM 只评语义。"

### 五、评测与迭代
- 评测 agent 自身用"人工标注集"校准:抽样 50 条,人工评分 vs agent 评分,算一致率(要求 > 0.8);
- 回归:评测 agent 升级后,旧 case 评分不能漂移;
- 指标:评测一致率、bad case 归因准确率、单次评测成本。

### 六、工程化
- CI 集成:每次 agent 改动自动跑评测,作为合并门禁(借鉴项目 5 个 CI job:lint/audit/test/load/migration-smoke)
- 可观测:评测结果写 Prometheus,看得分趋势(借鉴项目 Prometheus + Grafana 监控)

## 方向 C/D:批量生成 / 竞品监控(简要点)

### 批量生成 Agent
- **定位**:给定主题 + 约束,批量生成结构化内容(如商品描述、FAQ、话术)
- **MVP**:模板 prompt + 批量并发调用 + 质量过滤(规则 + LLM 抽检)
- **背书**:项目里 seed_data.py 批量生成种子数据、FAQ 知识库构建
- **风险**:agent 属性弱,容易做成"调 API 批量跑",要强调"质量过滤闭环"才有 agent 味

### 竞品监控 Agent
- **定位**:定时抓取竞品公开数据(定价、功能、评论),LLM 归纳变化 + 预警
- **MVP**:定时任务 + 抓取工具 + LLM 归纳 + diff 报告
- **背书**:项目里 usage_analytics / daily_report 的定时报告经验
- **风险**:agent 属性弱、偏数据工程;要强调"LLM 归纳异常 + 自动归因"才是 agent

## 附:通用回答框架(任一方向都适用)

无论选哪个方向,按这个 7 段式讲:

1. **定位**(15s):选哪个方向 + 一句话理由
2. **MVP 边界**(30s):做什么 + 明确不做什么(体现取舍)
3. **技术方案**(60s):架构图 + 核心模块 + 关键取舍
4. **实战背书**(45s):引用项目经验 + 数据指标
5. **怎么证明有效**(30s):评测指标 + 回归门禁
6. **工程化**(30s):测试、可观测、成本控制
7. **价值与延展**(15s):业务价值 + 四方向的统一性

总时长控制在 3-4 分钟,留时间给面试官追问。

## 附:高频追问预案

**Q:为什么不用现成的 prompt 优化工具(如 DSPy)?**
A:DSPy 适合学术研究,工程落地要考虑:可解释性(我的方案输出归因报告,人能看懂)、可控性(单轮 vs 多轮可选)、成本(小模型评测 + 强模型裁判)。MVP 先自建,验证假设后再考虑引入。

**Q:LLM-as-judge 本身不可靠怎么办?**
A:三点缓解:① 能用规则确定的绝不用 LLM(格式、字段、安全);② rubric 固化 + 多次采样投票;③ 抽样人工校准,要求一致率 > 0.8 才上线。本质上"评测"是概率工具,不是真理机器。

**Q:这个 agent 的 agent 属性体现在哪?**
A:它有工具(评测/归因/优化/裁判)、有循环(分析→优化→验证的闭环)、有记忆(历史优化记录做回归对比)、有自主决策(归因后决定优化方向)。不是单次 LLM 调用,而是工具编排的循环。

**Q:并发批量调用怎么保证安全?**
A:我有实战经验——项目里修过 3 个并发 bug:LRU 缓存竞态(加 RLock + 双重检查锁定)、SQLite 无 WAL(改 PRAGMA journal_mode=WAL)、幂等 TOCTOU(捕获 IntegrityError)。批量评测我会做:并发限流 + 超时重试 + 幂等键防重复。
