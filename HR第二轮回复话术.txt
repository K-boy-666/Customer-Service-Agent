**1、搭建一个AI agent需要做哪些工作？**

我认为搭建一个可用的 AI Agent，核心不是"调通 API"，而是做好六个环节：

- **场景与需求定义**：明确 Agent 解决什么痛点、边界在哪、MVP 最小可用范围是什么；
- **架构设计**：确定是单 Agent 还是多 Agent 编排，定义 Agent 间的通信协议和权限边界；
- **意图识别与路由**：让 Agent 能正确理解用户意图，并分发到对应的处理模块；
- **工具与能力设计**：给 Agent 配备可调用的工具（查询、创建、计算等），并控制权限；
- **记忆与状态管理**：支持多轮对话上下文、会话状态持久化；
- **评测与工程化**：建立可复现的测试集和评测指标，加上限流、日志、监控、CI/CD 等生产保障。

我在客服智能体项目里就是按这个流程从头走了一遍。

---

**2、使用过哪些搭建平台？这些平台有什么区别？**

我的个人项目是从零手写架构的，没有使用 Coze、Dify 这类低代码平台，核心用了 FastMCP 做 MCP Server 框架、SQLAlchemy + Alembic 做数据层、Docker Compose 做生产部署。

不过我对市面上的平台有了解：

- **Coze（扣子）**：字节出品，生态丰富，插件和知识库集成快，适合快速验证想法和做 C 端 Bot；
- **Dify**：开源，工作流编排能力强，适合需要复杂多步骤流转的企业级应用；
- **FastGPT / MaxKB**：侧重 RAG 知识库问答，搭建速度快；
- **LangChain / LlamaIndex**：偏代码框架，灵活度高，但需要自己处理编排和工程化。

我选择手写而非低代码平台，是因为项目目标是"准生产级"——需要完整的权限分层（RBAC）、并发安全、幂等控制、Prometheus 监控和 CI/CD 流水线，这些在平台上很难精细控制。手写虽然前期投入大，但对 Agent 全栈的理解更深。

**3、搭建过什么agent？效能如何？简要的实现原理是什么？**

我搭建的是一个**多 Agent 智能客服系统**，解决电商客服"咨询量激增、人工成本高、安全合规缺失"的痛点。

**实现原理**：
- 采用 **Orchestrator 单入口架构**，所有客户消息统一由 Orchestrator 接收，内部通过 Dispatcher 做意图识别后，路由到 6 个子 Agent（订单查询、售后处理、工单创建、投诉安抚、人工转接、咨询解答）；
- 意图识别用**规则引擎 + sentence-transformers 语义检索**双轨方案，支持 7 类意图并行识别和多意图拆解；
- 子 Agent 按 **L0/L1/L2 三级权限**分层：L0 只读、L1 可创建记录、L2 仅对话（防 prompt 注入）；
- Agent 间通信遵循固定协议：输入【客户上下文 + 任务】，输出【处理结果 + 客户回复 + 内部备注】；
- 数据层用 SQLAlchemy ORM，15 张表覆盖订单、工单、退换货、满意度、审计等完整业务；
- 工程化层面：slowapi 四层限流、structlog 结构化日志、Prometheus + Grafana 监控、GitHub Actions CI/CD。

**效能数据**：
- 84 个测试用例，76% 分支覆盖率；
- pytest-xdist 并行加速，测试耗时从 63s 降至 13s（5 倍提升）；
- 负载测试 P95 延迟 < 2 秒；
- 独立定位并修复 3 个生产级并发 Bug（LRU 缓存竞态、SQLite WAL 缺失、幂等 TOCTOU）。
